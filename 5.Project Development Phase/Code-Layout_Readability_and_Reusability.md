## Design Principles Applied

- **Single Responsibility** вЂ” each `src/` module does exactly one job
  (cleaning, one model, orchestration). `app.py` never talks to a model
  directly; it always goes through `predict.py`.
- **Fail-soft by construction** вЂ” `predict.get_models()` and
  `preprocessing.rule_based_predict()` guarantee the app never crashes due
  to missing model weights or a missing API key.
- **Unified schema** вЂ” both `model.predict_bilstm()` and
  `bert_model.predict_bert()` return the exact same dict shape
  (`emotion`, `confidence`, `emotion_scores`, `cleaned_text`, `model_used`),
  so `predict.py` and `app.py` don't need model-specific branching logic.
- **Constants over magic numbers** вЂ” thresholds (`MIXED_EMOTION_THRESHOLD`,
  `KEYWORD_SCORE_WEIGHT`, `BERT_CLASS_WEIGHTS`) live in `preprocessing.py`
  and are imported everywhere they're used, not duplicated.
- **Docstrings + inline comments** on every module explaining *why*, not
  just *what*, particularly around the fallback logic and keyword boosts.
- **Reusability** вЂ” `train.py` and `predict.py` both import the same
  `EMOTION_LABELS` / lexicon from `preprocessing.py`, so retraining on a
  new dataset automatically stays consistent with inference.

## Naming Conventions
- `snake_case` for functions/variables, `PascalCase`-free (no classes needed
  beyond the PyTorch `Dataset` in `train.py`).
- File names mirror their primary export (`bert_model.py` в†’ BERT-specific
  logic only).
